# Quality Guidelines

## Docker Build Best Practices

To ensure fast and efficient builds, we follow these practices:

### 1. Rust Backend Optimization
- **Use `cargo-chef`**: We split the build into a "planner" and "builder" stage. The builder stage "cooks" dependencies based on a `recipe.json` generated by the planner. This ensures that dependency compilation is cached in a Docker layer.
- **BuildKit Cache Mounts**: Use `--mount=type=cache` for `/usr/local/cargo/registry` and the project `target/` directory. This enables incremental compilation during development.
- **Multi-stage Builds**: Always use a multi-stage build to keep the final production image small. Use `debian:bookworm-slim` or similar for the runtime stage.

### 2. Frontend Optimization
- **pnpm Store Caching**: Use `--mount=type=cache,id=pnpm,target=/pnpm/store` during `pnpm install` to avoid re-downloading packages.
- **Frozen Lockfile**: Always use `pnpm install --frozen-lockfile` in CI/Docker to ensure reproducible builds.

### 3. CI/CD (GitHub Actions)
- **Multi-arch Support**: Build for both `linux/amd64` and `linux/arm64` using `docker/setup-qemu-action` and `docker/setup-buildx-action`.
- **Remote Caching**: Use `type=gha` cache for both `cache-from` and `cache-to` (with `mode=max`) in `docker/build-push-action`. This ensures intermediate layers are cached in GitHub's cloud.

## CI/CD Pipeline

- **Branch Strategy**:
  - `master`: Triggers a Docker build with the `edge` tag.
  - Tags (`v*`): Triggers a full release (Windows, macOS) and a Docker build with versioned and `latest` tags.
- **GitHub Packages**: All Docker images are hosted on `ghcr.io`.
