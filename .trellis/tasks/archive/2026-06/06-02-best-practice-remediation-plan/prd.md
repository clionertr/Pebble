# 最佳实践问题整改计划

## 目标

综合本会话中 Codex 扫描报告与用户提供的 DeepSeek 扫描报告，产出一份可执行的整改计划文件。计划需要同时列出两份报告暴露的问题，并为每类问题给出可验证、可验收的标准。

## 已知事实

- 用户要求使用 `grill-me`：需要逐步深挖计划边界，每次只问一个问题，直到关键点达成共识。
- 用户水平有限，希望指出不合理之处，说明计划需要解释取舍，而不是只罗列技术名词。
- Codex 报告发现的重点包括：
  - 前端依赖漏洞：Vitest、Vite、PostCSS、ws。
  - OAuth HTML 错误页未转义，存在反射型 XSS 风险。
  - 版本/仓库元数据漂移：`0.0.11` 与 `0.0.4`、`clionertr/Pebble` 与 `QingJ01/Pebble` 不一致。
  - Docker 基础镜像和 GitHub Actions 未做强 pin。
  - `.dockerignore` 漏掉本地 agent/config 目录。
  - nginx 信任所有 `X-Forwarded-For`。
  - CSP 与 GitHub 更新检查/OAuth inline script 存在冲突。
  - OpenAPI 与真实路由不同步。
  - 缺少 ESLint/Prettier、Rust audit/deny。
  - 双 lockfile、Trellis 规范占位、大文件/巨型模块、内部错误泄漏、本地运行数据留在仓库内、根目录大图等。
- DeepSeek 报告发现的重点包括：
  - RPC 层纯透传和薄包装过多，API 层出现业务逻辑泄漏。
  - HTTP 请求可达路径有若干 `.unwrap()`，关键错误被 `let _ =` 静默吞掉，错误类型不一致。
  - 类型、查询模式、`spawn_blocking`、CSV 解析和 Tantivy 命中构建存在重复。
  - 附件上传缺少请求体大小限制和速率限制，搜索端点缺少限流，session/rate-limit/OAuth state 仅内存且缺清理。
  - 命名、模块边界、可见性、注释语言、配置前缀不一致。
  - README、集成指南、SSE/API 文档存在过时或缺失。
  - OAuth、Compose、IMAP、搜索、API 路由、E2E 等测试覆盖缺口。

## 临时假设

- 这次先产出“整改计划文件”，不直接修代码。
- 最终计划文件建议放在 `docs/best-practice-remediation-plan.md`，便于团队阅读和版本化；本 PRD 仅作为 Trellis 工作过程记录。
- 计划应优先按风险和依赖顺序组织，而不是按报告原始章节机械排序。

## 待确认问题

- 已确认：最终计划文件面向“开发执行版”，重点是可拆任务、可验收、可落地，不做管理汇报风格。
- 已确认：两份报告的问题全部纳入计划，但先分阶段处理：第 0 阶段核验报告，第 1 阶段安全/稳定性，第 2 阶段契约/文档/工具链，第 3 阶段架构清债。

## 需求（演进中）

- 文件必须包含“两份报告暴露的问题”。
- 最终文件采用“开发执行版”：每个阶段说明改哪些文件/模块、为何优先、如何验收。
- 文件必须包含“具体的验收标准”。
- 验收标准必须能对应到暴露的问题。
- 文件需要指出用户输入或报告中不合理/可能过度修复的地方。
- 语言使用中文，表达尽量通俗但专业。

## 验收标准（演进中）

- [ ] 文件中有“问题总览/问题清单”，覆盖 Codex 报告与 DeepSeek 报告的主要发现。
- [ ] 文件中有“整改阶段/优先级”，说明先修什么、后修什么、为什么；阶段采用：0 核验、1 安全/稳定性、2 契约/文档/工具链、3 架构清债。
- [ ] 每个整改项都有可验证验收标准，例如命令、测试、文件检查或行为验证。
- [ ] 文件中单独列出“需要校验或不建议照单全收的 DeepSeek/Codex 结论”。
- [ ] 文件末尾有执行顺序建议，能指导后续拆任务。

## 暂不包含

- 本轮不直接修改业务代码。
- 本轮不承诺一次性修完所有问题。
- 本轮不做外部网页调研，除非后续用户要求依赖最新行业建议或 CVE 细节。
